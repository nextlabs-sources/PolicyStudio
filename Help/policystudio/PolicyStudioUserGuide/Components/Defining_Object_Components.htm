Defining Object Components

Regardless of which type of object component you want to define, the proce­dure is basically similar. For this reason, we’ll describe in a general way the common procedure shared by all six types, then examine the particulars of each type in more detail. Again, since action components are quite different, we will treat them as a separate topic.

The discussion of object components is divided into the following topics:

·        Common Procedure

·        Defining Document Components

·        Defining User Components

·        Defining Computer Components

·        Defining Application Components

·        Defining Portal Components

·        Defining Server Components

Common Procedure

To define any type of object component:

  1. Expand the appropriate panel to display existing components of that type. For Subjects, click on the tab you want: Users, Computers or Applications.

  2. Click the New button.

  3. Type a name for the component, and click OK. Names cannot be longer than 128 characters, and cannot include the reserved characters $, /, ?, \, &, or *. This opens the new component in the editing pane.

  4. In the Description field, provide a description of this component. Although this is not technically required, it is helpful to make this as descriptive as possible. It is also useful for recording the reasons for later changes to the definition of this component, if any.

About Membership and Properties

Besides a name and description, all object components are defined by one or more memberships, and one or more properties. This is one of the characteris­tics that distinguish them from action components, which have neither member­ship nor properties.

Membership identifies a superset or parent class of entities, of which this com­ponent is a subset. Properties are the metadata filters that define the subset. Thus: Component N = All members of parent class X that have properties A, B, and C. For example, a user component called Boston Staff = all members of par­ent class “All Employees” whose Status is “Current” and Location is “Boston.”

This is the most basic and common model for defining objects: to use the pre­defined “All [whatever]” parent component for the membership, and then spec­ify one or more properties to narrow down the definition.

 

The component editing pane

Defining Membership

The membership of a component will depend on what type of object it is. As a rule, it can be either an individual entity of the appropriate type (for example, an LDAP user group, for a user component), which you can either type in the field or select using the Lookup button; or an already defined component, which you can drag over from the component bin.

For the latter case, you will most often use one of the predefined “All” compo­nents: All Users, All Computers, and so on. If you like, however, you can also select a component that has already been defined, and further narrow it based on other properties.

If you do not define any membership for a component but only assign a property, the effect is the same as one of the “All” components; e.g., “All users with Location = Boston.”

To define a component’s membership,

  1. In the Operator field, choose one of the available operators:

    2. ·        In: The component represents a subset of the parent class or classes spec­ified here.

    ·        Not in: The component represents entities outside the parent class or classes specified here.

  2. To define the parent class, you have several alternatives:

    3. ·        Drag an existing component from the component bin on the left and drop it into this field. This parent component must be of the same type as the new component you are defining.

    ·        Type the name of an existing component.

    ·        Click the Lookup button to display all existing Information Network Direc­tory entities, and choose one or more from that list. (See "Using the Lookup Feature" below.)

  3. You have the option of leaving the Members field blank. If you do that, the membership will be All or Any—in our example, All Users.

Multiple Memberships

You can specify as many memberships as you like for each component. To define multiple OR memberships, drag more than one already defined component into the Membership field, or click the New Component line and type additional memberships into the open editing field. You do this to make the component’s membership broader or more flexible; it represents a logical OR expression: “Employees or Contractors with the properties ...”

To add multiple AND/BUT memberships, click the  icon to add a new field. You do this to make the component’s membership more restrictive—for exam­ple, a group of users who are both interns and foreign nationals: “In Interns” and “In Foreign Persons.” (A BUT expression is the same, but with a Not In operator—“In HR” but “Not In Contractors”.) You can add as many membership condition fields to a component as you like; just remember that they are logical ANDs or BUTs—they will all have to hold true.

Using the Lookup Feature

If you click the Lookup button, a dialog box pops up, allowing you to choose users, computers, applications, or documents, depending on which type of com­ponent you are defining. (For an example, see Using the Add Members Lookup tool.) Items in the Users and Computers lists are imported from Active Direc­tory; the icon next to each item identifies its source. For Document compo­nents, the lookup dialog contains a clickable folder and directory viewer that works like a standard browsing window. For Applications, the lookup window lists all applications that have been enrolled.

Note: For portal contents, the lookup feature supports only SharePoint por­tals. For other types, you must type the portal elements you want to add.

Membership Specification Syntax

Rather than use the lookup feature, you can manually type the membership for a component. When you do this, you are actually typing the name of a compo­nent. If no such component has been defined yet, Policy Author asks if you want to create it now. If you click Yes, a new component by that name will be cre­ated, but it is as yet undefined. (This is identical to the way you can create new components on the fly inside policies, which we will discuss in the following chapter.)

Object Component Properties

You use the Properties controls to specify the criteria that define the component as a subset of its parent class. Properties may be thought of as one or more fil­ters that operate on the metadata or other characteristics of the parent class. Specifically, you select a property from the list in the combo-box, type in a spe­cific value for it, and combine them with an appropriate operator:

·        String properties: is or Is not

·        Date properties: on or after or before

·        Number properties: “=”, “>”, “>=”, “<”, “<=”, or “!=” 

The available properties differ depending on which kind of component you are defining. For example, for a User component, you might specify Job Title Is Vice President; for a Computer component you might select Site Is Not Boston. Spe­cific properties for each component type are described later in this chapter.

Different types of object components have different default properties, but for Users and Computers, the list can be customized to include any properties defined in your Active Directory. If your administrator imports additional Active Directory attributes, these attributes appear in the Property Name drop-down list.

 

Object component: Defining properties

Available Properties

As mentioned earlier, the properties available in the drop-down list for any object component depend on the component type. For all component types, these lists include a set of default properties, which are always available because of Windows standards. The default properties for each type of object component are listed in their respective sections, later in this chapter.

In addition, there may be one or more custom properties available as well, if any have been defined by your system administrator. What these are depend on the component type. For example, for document components, the properties list may include custom document tags, such as “Internal Use Only” or “Extremely Sensitive,” which are useful for identifying documents for policy enforcement purposes.

Defining Properties

To define an object component’s properties:

  1. Select the property name from the combo-box. This list displays all default properties as well as any custom ones that may have been added.

  2. Select an operator from the list in the combo-box. These operators vary depending on what type of component you are defining.

  3. Type the value to be compared to the current property value. If the expression is true for a particular person, file, etc., then the policy is enforced. Note that you must have some knowledge of the actual values defined in your LDAP source, since you type a value rather than look up a value.

    4. Note: All property values are treated as case-insensitive.

  4. To verify that the definition you created reflects the items you intended, you can use the Preview command in the Window menu to display the Pre­view pane, then click the Go button. The Preview pane displays all of the individual entities that would be included in the component as it is pres­ently defined.

This feature is useful for testing or debugging components. If the list con­tains entities you did not expect or intend to include in the component you are defining, or does not contain entities you wanted to include, you know there is a problem with the definition of the component, and you can correct it before deployment.

Note: The Preview feature is only available for the object component types—not for action components, or for policies.

Defining Document Components

Document-type components share all of the generic object component charac­teristics discussed above. In addition, they have some special qualities as well, and they can be expanded to include custom attributes depending on your requirements.

Note that document components refers specifically to Windows or Linux file sys­tem documents. Documents stored on SharePoint or other collaboration portals are defined in a special way, as portal content components (see Defining Portal Components).

Note also that all policies that are enforced by the Removable Device Enforcer rely on document components that are defined in a special way. (This enforcer is available only with the Enterprise Data Protection product. For details, refer to that product’s design guide, Designing Endpoint Protection Policies.)

Document Components: Membership

To work with Document components, expand the Resources component bin, and click on the Documents tab. Click the New button to create a new Document component.

You define the membership of a document component in any of the following ways:

·        An existing document component, either dragged from the component bin or typed directly in the membership field

·        A document name and/or path typed directly in the membership field

·        A network path selected from the list displayed by the Lookup button, to the right of the Membership field

You can use wildcard characters in the file name or the path. The following table lists and describes all the supported wildcards. Note that these wildcards are also available for use in Portal components.

Wildcard Character

Description

*

Matches zero or more characters, not including the folder separator

**

Matches zero or more characters, including the folder separator

d

Matches any single-digit numeral, 0-9

D

Matches any numeral, one or more digits

a

Matches any single letter, either upper or lower case

A

Matches one or more letters, either upper or lower case

c

Matches any single alphanumeric character or special character, such as %, &, _, etc., except for backslashes, which are reserved

C

Matches one or more alphanumeric characters or special characters, such as %, &, _, etc., except for backslashes, which are reserved

s

Matches a single space

S

Matches one or more spaces

With the exception of the standard * and **, all wildcards consist of a letter pre­ceded by an escape character. Two escape characters are supported, a question mark and an exclamation point:

?<wildcard> means Matches the wildcard. It is useful in all kinds of general matching policies to existing files.

!<wildcard> means Does Not Match the wildcard. Beyond general matching pur­poses, it can be useful in enforcing that names of newly-created files conform to required conventions.

You can also use the escape characters for themselves. That is, if you need to specify a file name that contains a literal question mark, you can represent it as “??”. Similarly, for a literal exclamation point, use “!!”.

Different letters represent different kinds of characters, and each has two ver­sions: lower-case stands for a single character of a certain type, and uppercase stands for one or more of those characters. This wildcard convention enables you to define wildcard-based document components with a great deal of preci­sion. The following are some examples:

·        c:\Program files\* matches any file in the c:\Program files\ folder.

·        c:\Program files\** matches any file in the c:\Program files\ folder or any of its subfolders.

·        **.txt matches any local file with a .txt extension, regardless of its place in the directory tree.

·        **\secret* matches any local file with a name that starts with “secret”

·        **secret* matches any local file with a name that contains the word “secret” anywhere. A more natural way to write the same thing is **\*secret*, but the two expressions are equivalent.

·        **\Q?drevenue.doc matches any local file with a name that starts with “Q”, ends with “revenue.doc”, and has exactly one single-digit number in between. For example, \\documents\accounting\Q2revenue.doc would match, while \\documents\accounting\Q4-98revenue.doc would not match.

·        **\?d?d?d*?d?d*?d?d?d/d*.* matches any file that has a social security number anywhere in its name, either with hyphens or without.

·        ?c?c?c.* matches any file with name containing exactly three characters of any kind, in either upper or lower case.

Lastly, it is important to note that these special wildcards are available only for document and portal components—not for any other uses in Control Center, such as specifying e-mail addresses or domains, defining subject components, or fil­tering queries in Reporter.

About Content Analysis Policies

The last two properties listed in the table are used only in the context of Control Center’s content analysis feature. This feature lets you define policies that depend on the presence or absence of certain specified text content within doc­uments, e-mail subject line, or e-mail body. For example, you could define a component that represents any document containing more than five credit card numbers, say, or even a single instance of the keyword phrase Confidential Internal Data. You could then use this component in a policy that blocks any user from copying such documents onto any public server.

Content analysis policies can also invoke a special obligation that automatically redacts some specified content when it is discovered—that is, replaces the con­tent with a string of Xs—but allows the action to proceed in redacted form. For example, a policy might check all outgoing e-mails for account numbers, and redact them before sending the messages.

The Match Count attribute of the Contains Content property specifies the mini­mum number of instances a specified content string must be present in the doc­ument or e-mail, before the component (and thus the policy) applies.

For a detailed discussion of how you can use the content analysis feature in pol­icies, refer to the Policy Design Guide.

About the Include Only Directories Property

Document components have a special property, Include Only Directories, that allows you to control how policies with the component in them will handle directories, rather than files. This is a useful feature because although a policy can control all files in a certain directory, it cannot stop a user from copying or moving a parent directory, thus circumventing the policy. This feature allows you to write policies that prevent this and similar problems. For more details, see the Policy Design Guide.

When you define an application component, it is available for use as a part of the subject in policies. Bear in mind that applications are logically connected to the user component and the computer component; all these three work together as the single subject of the policy.

It is important to understand that application components refer specifically to applications, not to any file type or extension that may be associated with applications. For example, you might create an application component called MS Word. If you then use this component in a Deny policy, the policy blocks users from using Word to open the specified documents. Although Word is asso­ciated with a .DOC file extension, this does not mean that this policy applies to .DOC files generally; the policy does not block anyone from opening .DOC docu­ments using another application, such as Notepad or WordPad. Again, the com­ponent refers to the application, not the file type.

Also, bear in mind that Control Center considers different versions of an appli­cation as entirely independent of one another, and they must be enrolled sepa­rately. If you have more than one version of an application running in your environment and enrolled in Control Center and you want to create application components to use them in policies, you must create a separate application component for each, and make sure they are specified in the policies you write.

Predefined Application Components

Control Center provides several predefined application components that allow you to easily set up commonly desired policies. To use the predefined compo­nents, you must first import the seed data for your Data Protection product. For information about importing seed data, see the Control Center Administrator’s Guide.

Autorun Prevention

In many cases, security administrators may want to block any executables from automatically launching on network PCs. There are two predefined application components that can help with this:

 

·        Autorun.exe represents the autorun executable. You can create a policy with this application as the subject, to prevent autorun from launching anywhere when a CD or other removable media is inserted or attached to a PC: “DENY All users, on all hosts, using Autorun.exe, for all actions, on all documents.” This component is read-only.

·        Applications that Start Automatically is a more generic component that you can define yourself. It contains the Autorun.exe component, but also allows you to add any other applications you have enrolled, that you want to be able to block.

Application Monitoring

Unless you specify otherwise, Control Center applies each policy to all applica­tions running on monitored PCs. However, there are two other predefined appli­cation components that allow you to define a list of applications you want to either exempt from monitoring, or explicitly include for monitoring, while ignor­ing all others.

 

·        Monitor All Applications—Exceptions: If you want to exclude specific applications while applying to all others, add them to this list and deploy this component.

·        Monitor Specific Applications: If you want your policies to monitor only specific applications and ignore all others, add them to this list and deploy this component.

They appear in the list in the application component bin, just like any other application component you might define. However, they work in a different way: they are global for all policies—that is, once defined and deployed, they apply to all policies deployed in your network. They have the same effect as if you had included an application component in every deployed policy, so that it only works on some applications and ignores others.

You can define one or the other of these components (or neither of them), but you generally should not define both. Note that these two function as described on their own, simply as components. That is, you deploy them by themselves—there is no need to use them in any policy.

By default, Control Center monitors all applications on desktops where Desktop Enforcers are running. If you do not define a list for either of these application components, this is the expected behavior.

If you specify one or more applications to monitor, then all Desktop Enforcers monitor only those for all deployed policies, and ignores all other applications. This may be useful if you are launching a limited set of policies that you know involve only a well-defined set of applications.

On the other hand, if you define one or more exceptions, Control Center will monitor all applications on desktops where Desktop Enforcers are running, with the exception of the applications you specify. You may use this feature if you have some application—for example, an anti-virus security tool running on every desktop in your network—that you are sure you will never want any policy to block for any user under any circumstances.

Note: For best system performance, you should define exceptions for all operational and support software, for example, anti-virus scanners, back-up software, disk defragmentation software, and search indexes.

If you do define and deploy both a Monitor Specific and an Exception application component, the latter will take priority over the former. That is, only those applications specified in the Monitor Specific list are monitored, with the excep­tion of any that are listed in the Exceptions list.

Note: You should not delete either of these predefined components for any reason. If you do delete one, you cannot simply recreate a new com­ponent with the same name; it will not function properly.

Windows Explorer

Note that for the purposes of excluding or including, Windows Explorer is consid­ered an application if you are explicitly excluding applications, but not if you are explicitly including applications. That is, if you deploy the Monitor Specific Applications component, Windows Explorer will be monitored by default even if you do not include it in the list. In this case, it will be considered a part of the operating system rather than an application. As a result, Control Center will enforce policies based on actions such as move, copy, delete, save, and so on.

On the other hand, if you deploy a Monitor All Applications—Exceptions compo­nent, you can list Explorer as an application. In this case, Control Center treats Explorer as an application, and will not enforce policies governing any of Explorer’s file handling functions. You should be very careful about doing this, since it will override any policies that include standard file-handling functions—copy, move, delete, save, etc.

Controlling Web Applications

If you need to design policies that control the use of web applications, you can do so by creating application components with the URL property specified as one or more URLs where the web applications are hosted. (In fact, all such applica­tions will be based on Internet Explorer, but this does not need to be explicit in the component’s definition.)

To use a common example, you may want to block users from attaching sensitive files to e-mail via the popular services G-Mail and Yahoo Mail. To do this,

  1. Create a new application component, and name it Webmail Applications.

  2. For the Member field, drag All Applications from the component bin, or leave it blank.

  3. In the Property panel, select “URL” from the Property Name combo-box list, and type http://mail.yahoo.com/** OR http://www.gmail.com/** for the Property Value. This creates an application component that represents both these blacklisted web applications.

  4. Create your policy, based on this application and the Run action:

For “Sensitive Data”
On Run
By All Users using Webmail Applications
Do Deny and Notify “Sending sensitive information using web mail application is prohibited.”

(Sensitive Data is a document component that is already defined.) Once this policy is deployed, all users are still able to open the G-Mail or Yahoo Mail web­sites, and send e-mails without attachments, or with non-sensitive documents attached. However, if they try to use either site to send a document covered by the Sensitive Data component, the policy blocks the action and displays the notification message.

Note that the above is just an example; you can create any URL-based applica­tion components you like, based on any URLs you want to block, monitor, or oth­erwise base policies on.

It is helpful to think of device components as a path, since they are recognizable as a drive location on the local PC. Syntax of Device Properties

Bear in mind the following points about the syntax of the properties you specify for device components:

·        The Vendor and Product IDs will always be four-character hexadecimal values, rather than actual names like “Apple” or “iPod”. (For help finding these for a given device, refer to the Control Center design guide.)

·        All property values are case-insensitive.

·        All delimiters must be forward slashes.

·        When using wildcard characters, pay attention to the number of levels you really need to define for a component to behave exactly as expected. For example, if you want a component to represent all USB storage devices and you define it as device://USB/**, the definition is too broad: it will cover not only storage devices, but also other types of USB devices such as network cards, which will cause undesirable results. (Bear in mind, USB is used by many connections inside PCs—it does not refer only to the exter­nal ports where you plug in peripherals.) In this example, the definition should include another level, device://USB/storage/**, to cover only the intended devices.

Defining Portal Components

Portal components represent areas on a collaboration portal such as SharePoint. They resemble path-based document components in that they can represent places on the portal where you may want to control access to specific users, or places whose content you want to control in some way. You can use portal com­ponents to define controlled areas on a collaboration portal, just as you use doc­ument components to define controlled directories on the file system.

Specifically, portal content components can represent the following kinds of entities:

·        Portals

·        Sites

·        Web pages

·        Portlets (also called Web parts): libraries, lists, discussions, calendars and other structure elements smaller than sites or pages

·        Library items

·        List items

To work with portal content components, expand the Resources component bin and click on the Portal Content tab there. Click the New button to create a new portal content component.

Portal Content: Memberships

As with all object components, you have the option of defining portal content by typing an element name into the Membership field or dragging an existing component into that field. Wildcard characters that are supported for Document components are also supported for Portal components. See Wildcards for Docu­ment and Portal Components.

Note: As with document components, if you type the item manually, you must type the precise path and name, because there is no value check mechanism to detect errors. The most reliable way to define these components is to open the SharePoint area or item in your browser, then copy the URL from the browser into this field in the component definition.

Here are some examples of the required syntax for different types of SharePoint content. All examples represent items on a SharePoint 2007 server called blue­jungle:

·        For a site called “Alpha”:

//sharepoint2007.bluejungle.com/Alpha

·        For a sub-site called “abc” and all its children:

//sharepoint2007.bluejungle.com/Alpha/abc/**

·        For all items in a library “Docs 2”:

SharePoint://sharepoint2007.bluejungle.com/Alpha/Docs 2/**

·        For a list “DealList”:

SharePoint://sharepoint2007.bluejungle.com/Alpha/Lists/DealList

·        For a library item “IND Design.doc”:

SharePoint://sharepoint2007.bluejungle.com/Alpha/Docs 2/IND Design.doc

·        For a list item called “Test1”:

SharePoint://sharepoint2007.bluejungle.com /Alpha/Lists/DealList/Test1

Defining Server Components

The last category of object components, Servers, is used only in policies designed for the Enforcer for Windows Networks. These components let you write policies based on transfer protocols, by blocking communication across specified communication ports.This can be useful for blocking, restricting or monitoring activity over protocols that rely on standard port numbers—for example FTP (20, 21), Telnet (23), HTTP (80), HTTPS (443), or any other stan­dard communication protocol you want to control or monitor.

Because the Network Enforcer runs on the desktop, it is a part of the Enterprise Data Protection product. For more details, refer to the policy design guide for that product, Designing Enterprise Data Protection Policies.
