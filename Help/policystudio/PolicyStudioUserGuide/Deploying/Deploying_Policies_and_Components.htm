Deploying Policies and Components

Once a policy or policy component is created, it must be distributed to the Pol­icy Enforcers on all PCs or servers where policy enforcement is desired. An orga­nization may have specific requirements for controlling changes to computer systems, so one person is typically assigned to control access to policies and the distribution of policy changes to enforcement points. The process of distributing and organizing the distribution of policies and policy components is called deployment.

The Deployment Sequence

The deployment process can be broken down into a sequence of several tasks:

  1. Checking Dependencies: see whether deployment depends on comple­tion of other components.

  2. Setting Deployment Targets: specify which machines should receive the deployment. For DAC policies, this step is required, and must be done before submitting policies. For other types of policies, this step is optional because of the default setting, Auto Deployment, in Control Center.

  3. Submitting for Deployment: place a policy or component into the list of objects awaiting deployment.

  4. Scheduling Deployment: specify when deployment should occur.

  5. Confirming Deployment: check to see that deployment occurred accord­ing to the specified targets and schedule.

Each of these tasks is described in more detail later in this section.

Deploying All

If you have submitted several policies or components for deployment, you can use the Deploy All feature to deploy them all at the same time. This feature is also useful if you want to use Control Center’s default deployment targets and deployment time (immediate deployment). You can also deselect some of the components and policies if you do not want to include them in the deployment.

To deploy some or all policies and policy components that have been submitted for deployment:

  1. Select Actions > Deploy All. Control Center identifies all policies and pol­icy components that have been submitted but not yet scheduled for deployment. Control Center performs a dependency check to see whether there are any supporting policy components that you must finish and sub­mit first. If everything is finished and submitted, the Deploy All window appears with all submitted policies and components selected.

    2.  

    The Deploy All window

  2. Uncheck any items that you don’t want to deploy with this batch. You can only deselect an item if no other policy object being deployed depends on it.

  3. Click OK. The Deploy window appears, with all dependent components checked.

  4. For Deployment Start Time, choose one of the following:

    5. ·        Deploy Immediately (the default option): Deploys the policies immedi­ately, in Push Deploy mode, that is, without waiting to receive a heartbeat message (see About Push Deployment).

    ·        Deploy at <date and time>: Starts deployment at the default date and time, as displayed. By default this is set at midnight of the current day, but you can set a different time if you wish. (This might be delayed slightly by the heartbeat mechanism, which is set to one hour by default.)

    ·        Specify date and time controls: Display the current time by default, but you can change them to set any specific moment you want deployment to begin.

  5. Click OK. The policies and components are automatically deployed to all required deployment targets, as identified by Control Center. For infor­mation on setting targets manually, see Setting deployment targets manu­ally.

 

The Deploy window

The Full Deployment Sequence

As mentioned previously, the full deployment sequence—as distinct from the quick deployment described above—comprises five steps. This section describes the steps in detail.

1. Checking Dependencies

Whenever you finish defining a component or constructing a policy and are ready to deploy it, you should check whether it depends on the completion of other components. Control Center automatically performs a dependency check when you submit an object for deployment, but you can also perform this check yourself at any time.

  1. In the navigation pane, or using the tabs along the top of the editing pane (if you are editing multiple objects at the same time), select the policy or policy component whose dependencies you want to check. If you are edit­ing a policy or component, it is already selected.

  2. Select Actions > Check Dependencies. If dependencies are found, the Check Dependencies dialog box appears with the results. For each depen­dent component, an explanation of the dependency is shown.

  3. Read the information in the dialog box. If it shows that you must take action, such as finish editing another component before you can deploy the current one, make a note of the required task. If the dependent com­ponent is greyed out, you do not have permissions to perform the required task yourself. The dependent component can be in the states described in the following table.

    State

    Description

    Missing

    The component has never been submitted for deployment. If it is required by the object you are trying to deploy, you will need to submit the missing component first.

    Required

    The component has been submitted for deployment and will automatically be deployed along with the current component.

    Modified

    An existing version of the component has been deployed, but someone has created an edited version more recently that has not yet been deployed. The existing deployed version will be used until the modified version is deployed.

    4. However, if you want your new object to use the more recently modified version, you can open that object, see why it has not yet been deployed, and move the process forward if possible. For example, perhaps the modi­fied component needs to be submitted for deployment; if it has been sub­mitted, perhaps it has not yet been scheduled; or perhaps it has been scheduled, but its deployment time is too far in the future.

  4. When you finish reading the information, click OK.

2. Setting Deployment Targets

Before a policy or component is submitted for deployment, you can specify which Policy Enforcers you want it deployed to. To do this,

  1. Select Action > Set Deployment Targets. The Set Deployment Targets dia­log box appears.

    2.  

    Set Deployment Target dialog

  2. In the Deployment Type drop-down list, choose one of the following:

    3. ·        Auto deployment: Control Center automatically chooses the machines—all qualified endpoints—that should receive the policies. (This is the default behavior.)

    ·        Manual Deployment: You can specify which hosts you want to receive the policies.

    If you choose Manual Deployment, the window expands as shown below. You can use the File Servers, Desktops, Portals, and Active Directory tabs in the left-hand pane to specify which enforcers of each type you want this policy deployed to. Each tab displays a list of groups of enforcer hosts or individual host names. (The File Servers tab will show both Windows and Linux hosts, without distinc­tion.) If no deployment targets have been set up yet, this list is blank.

     

    Setting deployment targets manually

  3. You build your deployment target list in the right-hand pane, by selecting an enforcer on one of the tabs, then clicking the Add button.

  4. To delete a deployment target from the list, select the desired group or machine name and click Remove.

    5. Note: You cannot select items within groups, only the group as a whole.

  5. When finished, click Save.

3. Submitting for Deployment

Submitting a policy or policy component for deployment places it in a state where it is ready to be sent to policy enforcers and take effect within your orga­nization, but it is not deployed until someone schedules the actual deployment. You can select multiple objects for submission.

  1. In Policy Author, select the policies or components you want to deploy.

  2. Select Action > Submit. Alternatively, if you are submitting a policy or component that is open, you can click the Submit button.

    3. Control Center automatically performs a dependency check to see whether any additional policy components must be finished and deployed before you can continue. If there are no dependencies, you are ready to schedule deployment.

  3. If Control Center discovers one or more dependencies, it displays a dialog box describing which new or modified components are involved. You will not be able to proceed with deployment until these components have also been submitted; they will display as Missing.

4. Scheduling Deployment

After a policy or component is submitted for deployment, you can specify when deployment should occur.

  1. In the navigation pane, or using the tabs along the top of the editing pane (if you are editing multiple objects at the same time), select the policy or policy component for which you want to schedule deployment.

  2. Click Deploy. Control Center performs a dependency check. If it finds that the deployment depends on the completion of other policy components, a dialog box appears to explain why the deployment could not be com­pleted.

    3. If there are no dependent components that require attention before deployment, the Deploy dialog box appears. This dialog box shows which policies and components will be deployed: the one you are working on, plus any dependent components detected in the dependency check.

  3. For Deployment Start Time, choose one of the following:

    4. ·        Deploy Immediately (the default) deploys the policy right away, without waiting for heartbeat messages from the enforcers. This is called push deployment; for more details see below.

    ·        Deploy at <date and time>: Starts deployment at the default date and time, as displayed. By default this is set to midnight of the current day, but you can change it if you wish. If you set it to deploy now, there may be a short delay due to the heartbeat mechanism, which by default is set to one hour.

    ·        Specify date and time controls: displays the current time by default, but you can use the calendar and time widgets to set any specific minute you want deployment to begin.

     

    Setting Deployment Start Time

  4. In Deployment Contents, look for the Modified Components category. If present, it shows policy components that are needed by the object you’re trying to deploy and that have been modified and resubmitted for deploy­ment. Check the box next to any of these components that you want to schedule for deployment at the same time.

  5. Click OK.

5. Confirming Deployments

Once you finish defining and deploying a policy or a component, you may want to confirm that it has indeed been deployed in the way you intended. In addi­tion, when you construct and deploy a large number of components and poli­cies, it is often helpful to be able view summary of where they are deployed.

Policy Author offers several convenient tools that allow you to monitor the sta­tus of deployed policies and components; these are described later.

About Push Deployment

As discussed previously, you can schedule a deployment to Deploy Immediately, which is called a push deployment. This feature is useful if you need to check the results of a policy right away, since it avoids the delay that might be caused by the heartbeat interval. By default that interval is set to one hour, but it may be much longer.

Push deployments are distributed to all enforcers where this feature is enabled. In response, the enforcers’ heartbeats are all reset, which means they are syn­chronized from that point on. In a production environment with tens of thou­sands of enforcers, such simultaneous heartbeat traffic can cause noticeable spikes in network load. For this reason, the Push Deployment feature is recom­mended for use only in test environments, or in relatively small-scale production settings—less than, say, a few thousand enforcers.

Note also that only enforcers that have this feature enabled will receive push deployments. The feature is enabled in the enforcer profile, and is disabled by default; for details, see the Administrator’s Guide.
