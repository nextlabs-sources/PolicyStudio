Overview

Policy Author is a graphical user interface that you use to accomplish the follow­ing tasks:

Modeling: This refers to the process of creating an abstract model representing the various parts of the enterprise environment, such as users, applications, documents, and desktop PCs. The actual work of modeling consists of defining policy components. A component is a named definition that represents a cate­gory or class of entities, such as users, data resources, or applications; or of actions, such as Open or Copy.

Components can be thought of as the parts of speech you use to construct your policy statements. For example, you might define “noun” policy components, such as All employees in the human resources department or Any file with an .xls extension, and “verb” components like Copy, Print, and Rename File.

Constructing Policies: This refers to the process of building information control policies in Control Center. Policies use components as building blocks to repre­sent rules that you wish to enforce to control information access and use in your organization.

For example, you might construct the following policy: “Allow only product managers to download or print spreadsheets.” The components used in this pol­icy are “product managers,” “download,” “print,” and “spreadsheets.” When finished, policies are stored in a Policy Master database for distribution to the network components that enforce them.

You can qualify policies using conditions. Conditions can change a policy’s effect based on dynamic comparisons, evaluations, or contextual factors. Policy Designers can select pre-defined Conditions, including the time of day, day of the week, connection type, and last heartbeat. More complex conditions can also be written directly in ACPL.

Finally, a policy also includes notification obligations, such as notifications to the end user or e-mail warnings to an administrator whenever a policy is enforced anywhere in the network.

Deployment: Deployment simply means the distribution of new or modified pol­icies and policy components to the appropriate enforcement points on desktop PCs, laptops, and file servers throughout the organization. This means you can create, review and refine policies as long as you like, but they are not enforced until you actually deploy them.

Versions of Policy Studio

There are two versions of Policy Studio: the Enterprise Edition, which includes the full functionality, and the Dynamic Access Control (DAC) Edition, which includes support only for deploying policies to Windows Active Directory. The basic Policy Author interface is the same for the two versions; however, there are differences in what components display. Using Policy Studio for DAC policies is discussed in more detail in Dynamic Access Control Policies.

User Permissions

A Policy Studio user can have full permissions to create and manage policies and policy components, or limited permissions. For example, some users might be assigned permissions to create policies and all types of policy components, some might have permissions to create policies but not components, while others might be able to view policies only. User permissions are defined in the Adminis­trator application by the Control Center administrator.

About Views

For the Enterprise version of Policy Studio, you can configure views to control which UI features are displayed. You might do this if you only have certain kinds of enforcers installed (SharePoint, for example) and only want to see UI features for that enforcer. The three views are:

·        File System View: All controls and features relevant only to file system-based components are displayed and active. All those relevant only to col­laboration portals—the Portals component pane, for example—are hidden.

·        Portal View: All controls and features relevant to collaboration portal-based components are displayed and active to use. All those relevant only to file systems—the Attach to E-Mail and Copy Content basic action, for example—are hidden.

·        Corporate View: All controls for all component types are displayed and active.

The default view is Corporate. Note, however, that if you do not see some of the features described in this manual, it might be because you are working in one of the other more restricted views.

Starting Policy Studio

Policy Studio is installed as a standard Windows application; you launch it by selecting Start > All Programs > NextLabs > Policy Studio. When you start Pol­icy Studio, the Policy Author interface displays by default. You can toggle between the two interfaces using Ctrl + Tab, or the Policy Author and Policy Manager commands in the Window menu.

Logging In

When you start Policy Studio, you must specify a user name and password and a Policy Management Server host name. The user name can be any user currently defined with permission to use Policy Author. The default user, Administrator, has a password assigned during the Control Center installation; other users are defined by the system administrator after installation.

 

Logging In to Policy Studio

The Policy Management Server is the machine where the Control Center you want to use is running; this is specified during Policy Studio installation, and dis­plays automatically. (The Management Server is a software component of Control Center.) You also have the option of connecting to any other instance of Control Center, as long as you know its host name and the valid user name and password for it. Policy Studio displays all currently running Control Center serv­ers it can detect, in the Policy Management Server combo-box; you can select the one you want to use.

Logging Out

To close Policy Author, use the Exit command in the File menu, or click the stan­dard Windows X icon in the upper right corner of the window. Your work is always saved automatically when you exit the application.

Menu Commands

There are six menus available in Policy Author: File, Edit, Tools, Actions, Win­dow, and Help. The table below describes the functions of all commands in these menus. Note that several of the commands may be disabled (greyed-out) at any given time, depending on whether they are logically valid for the type and state of the currently selected policy object. For example, if no object is selected, the Properties command is disabled; or if the selected policy is not currently active, the Deactivate command is disabled.

Some of the menu commands, such as Delete, Submit, and Deactivate, can be applied to multiple policy objects. To select multiple policies or components, press Ctrl and click each object. All the selected objects, however, must be in the appropriate state for the command to be enabled. For example, to delete multiple objects, the selected objects must either be in Draft (This object has not been deployed) or Inactive status; in other words, all the objects must be in a state in which they can be deleted.

Interface Panes

Defining Components in this manual provides a detailed description of how you use Policy Author to define components. This section provides a brief overview of the tools available in Policy Author.

 

Policy Author views

As the figure shows, the Main window is divided into three panes:

·        The policy tree, at the upper left, is where you store, organize, and retrieve policies once you have defined them.

·        The component panels, at the lower left, are where you store and retrieve components once you have defined them. To construct policies, you simply drag components from one of these panels into the appropriate fields in the editing pane. (Click on one of the panels to expand it, if it is collapsed.) Note that some of the panels are further organized into sev­eral tabs for different component types.

·        The editing pane, on the right, is where you work as you define new com­ponents or policies, or edit existing ones. You can use the buttons at the bottom of this pane to control the state of a policy: to submit, deploy, or modify it.

Each object you open is displayed on a tab in this editing pane, and stays open until you close it. Each tab within this pane can be docked; you can grab it and reposition it below or to the right or left of other tabs. This allows you to view more than one object at once in a tiled layout, as shown in the following figure.

The Preview Pane

A fourth pane, Preview, can be opened as needed, using the Preview command in the Window menu. The Preview pane allows you to display all of the individ­ual entities that would be included in a component as it is presently defined. It displays at the right side of the window.

 

Previewing component contents

This feature is very useful for testing or debugging components. If the list con­tains entities you did not expect or intend to be included in the component you are defining, or does not contain entities you wanted to include, you know there is a problem with the definition of the component, and you can correct it before deployment.

You can also use this feature to filter an existing component by one or more properties. For example, you can open All Users, under Properties select “Department Is Finance,” then click the Go button in the Preview pane. The resulting list will show a list of all the individual users in the Finance Depart­ment. This is very useful in determining which property is most suitable for defining new components, since it allows you to see the actual results of various definitions.

The preview feature is available only for User, Computer, Application, and Docu­ment components. For Document components, the preview generally displays a list of file names, but if you have included an Include Only Directories property, it also lists network directories. (For details on this feature, see About the Include Only Directories Property.)

Using Wildcards

When searching for components in the Preview Pane, you can use the wildcard characters ’?’ (match any single character) and ’*’ (match any one or more char­acters) to filter by properties. For example, you could filter Users by LastName Is W*, as shown in the figure below. Note that specifying the All Users in the Users field is not required; it could just as well be left blank.
