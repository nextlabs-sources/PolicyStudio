About Communication Policies

The discussion throughout this chapter reflects the default policy type, Docu­ment policies. (As we mentioned above, you designate the policy type when you first create a new policy, on the same screen where you assign a name.) The other type, Communication policies, has several special features that are related to their ability to control e-mail attachments, instant messages, or other communications based on the recipients’ individual address or domain, as well as on any other available user property. Specifically, they have two special fields in the Policy Author editor, and also have an extended set of custom obli­gations.

The following is just a brief overview of communication policies. For more detailed discussion of how you can take advantage of their special features, refer to the policy design guide for whichever product you are installing.

 

Communication Policies: Special Features

Special Fields

There are two special fields available in communication policies: And Recipi­ents, and With Attachments. (The Using Channel field is just a different label for the Action component; it will always be one of the communication actions.)

The And Recipient Field

This field lets you specify one or more specific recipients of communications to which this policy will apply. Recipients are represented as User components; you can drag defined User components into this field, or you can type contents directly. If you leave this field blank, the policy will apply to all recipients, so if it is a Deny policy it will block users from sending the specified communications to any user.

In order to support this feature, User components have two special properties, E-mail Address and Domain. You can use them to define a user component based individual e-mail address or domain, just as you might use other properties like Location or Job Title. This is useful for creating user components representing, for example, an e-mail blacklist (or whitelist), which you then use as the And Recipient field of your communication policy.

Note that you can use wildcards in defining e-mail addresses as user properties, but not with domains. When defining users based on domains, you must supply one or more whole domain strings.

The With Attachments Field

Two of the communication-related basic actions, E-Mail and Instant Message, are somewhat special in that they can be used in two distinct kinds of policies: those that control attaching specified documents to e-mails or IMs, and those that control sending e-mail or instant messages at all.

If you create a policy based on either of these actions and do not specify any document components in the With Attachments field, then the policy will apply to sending any e-mail or IM, with or without an attachment. That is, the policy will focus on communication per se, rather than sending some specific class of data. For example, it might block certain users from sending any e-mail mes­sages to any user at a specified domain, regardless of whether the messages have attachments.

 

The With Attachments Field

By contrast, if a policy based on one of these actions does specify a With Attach­ments component, then it will be enforced only when that user tries to send an e-mail or IM to that recipient, with the specified kind of file attached to it. If it has some other file attached, or no attachment at all, then the policy will not apply to the e-mail or IM action. This means that, for example, a policy that prevents Sales staff from sending unapproved price lists to any customers, would still allow them to send approved price lists without restriction.

Special Obligations

Two special pre-defined obligations are available in Communication policies:

·        Disclaimer: When users initiate a chat session via Office Communicator, this obligation automatically attaches a configurable message (for example, a confidentiality warning) at the end of the first message in the session.

·        Participant Limit: This obligation counts the persons participating in a communication session, and warns the end user whenever the number is greater than some specified limit. When a user initiates a communica­tion session—chat, call, or web meeting—that includes too many partici­pants, this obligation displays a modal Windows message like the one shown above. This gives the user the choice between canceling the ses­sion, or continuing. It is valid only with the following actions:

o                   Send IM

o                   Send File (attached to IM)

o                   Voice Call

o                   Video Call

o                   Paste (from IM into another application)

o                   Meet (initiate meeting via Live Meeting)

Because these obligations are enforced by the Office Communicator Enforcer, they are only available to Entitlement Manager for SharePoint users. For details in using them, refer to the policy design guide for that product, Designing Endpoint Pro­tection Policies.
