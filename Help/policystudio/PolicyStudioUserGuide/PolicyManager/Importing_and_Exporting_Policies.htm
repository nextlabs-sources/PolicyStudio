Importing and Exporting Policies

Policy Manager provides a mechanism for importing and exporting policies and components, which lets you share policy objects you have defined in one instance of Policy Studio with other instances. This feature is very handy in col­laborative environments, for example, where policy construction responsibilities are distributed among several policy designers. It is also useful for transferring policies and components from an isolated development and test environment into your live production environment.

The mechanism works by saving defined objects from one Policy Studio database either in Active Control Policy Language (ACPL) or eXtensible Access Control Markup Language (XACML), depending on which kind of export you select. Once exported, ACPL files can be re-imported, or imported into another Policy Studio.  XACML files can be imported into other XACML-compliant Policy Management Systems.

Exporting Objects

With designated objects selected (for example, in the Policy tree), click on the Export icon to export policy objects. If you are exporting folders, select them in the tree and click the Export icon above the tree. If you are exporting individual policies, select one or more (use Shift + click or Alt + click to select more than one), then click the Export icon in the upper right of the Object Grid window.

Clicking on this icon launches the Exporting Policies window, which prompts you for the name and location you want to assign to the exported file. Once created, XML files can be imported into other instances of Policy Studio using the Import command or with the import.bat utility (see the Control Center Administrator’s Guide).

Importing Objects

To import policies and their required components into your Policy Studio, select File > Import. This command launches the Importing Policies window, which prompts you for the path and name of the file describing the objects you want to add to your object tree. You must specify this in the Import File Name field, or use the Browse button to browse for it. Note that only ACPL files (in .xml for­mat) can be imported into Policy Studio.

 

The Import window

These importable XML files can be created either directly from Policy Manager (see Exporting Objects), or using the export.bat utility (see the Control Center Administrator’s Guide).

Conflict Handling Options

When you import policies into Policy Studio, it is possible that there may already be existing policies or components with the same name. The import tool has four ways of handling this kind of name conflict, and you can choose the one you prefer by selecting its corresponding radio button in the Importing Policies win­dow as shown in the previous figure.

These include:

·        Resolve all name data conflicts by Overwriting the existing objects on the server with the data imported from the source file. The import-overwrite operation modifies existing policies and components on the server; there­fore, this operation requires that you have the same ability to modify pol­icies in Policy Studio. Specifically, the following conditions must be met:

o       You must have access control permissions to access, read, and mod­ify the policies and their components.

o       If a policy being imported has a different hierarchical structure from an existing policy, the overwrite operation attempts to delete any subpolicy in the existing policy that is not in the policy being imported. To delete a subpolicy, you must have delete permissions, and the subpolicy must be in draft or inactive state, and not sched­uled for future deployment, that is, the subpolicy must be in a state in which it can be deleted. ·        Resolve all data conflicts by adding the imported data to the server, with a new name generated by Adding a Suffix. You can specify the string you wish to use as a suffix; a date/time stamp is the default string, as shown.

·        Resolve all data conflicts by keeping the existing policies and components on the server, and ignoring the imported data.

·        In the event of any data conflict, cancel the import operation: do not change any of the currently defined objects, or import any new ones from the file.

About Shallow Import

The shallow import feature lets you perform a special kind of import: for any components or policies that are defined with reference to specific user or group names or to network hosts or paths, those references will be removed before importing. Only aspects of the definitions that do not tie the definitions to the physical environment—such as LDIF metadata and file or user properties—are included in the import. The original policy tree folder structure, and nested component structures (components whose definitions contain other compo­nents) are also preserved.

To perform this kind of import, check the Shallow Import check box.
