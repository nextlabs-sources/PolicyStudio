Monitoring Objects

This section presents a more workflow- or task-based perspective of how you use Policy Manager to monitor the components and policies deployed in your sys­tem.

Monitoring by State

The way this is done differs somewhat depending on which state the displayed objects are in—that is, which of the five State tabs is selected. For this reason, we will discuss the five State tabs each in turn.

Segregating Administrators

The assumption here is that responsibilities for designing and constructing the policies, and those for reviewing, deploying, and managing them after deploy­ment, are delegated to different administrative personnel. There is no way to assign permissions for users who can use Policy Author or Policy Manager only, but administrators can be segregated by function. For example, the Deploy per­mission can be assigned only to the administrator authorized to review and approve policies, but not to the administrator or designer who can only con­struct policies in Policy Author.

These permissions are assigned in the Administrator application by the adminis­trator who creates and manages users, groups, and roles.

Monitoring Draft Objects

The Draft state tab shows policies and their child components in the selected folder that have been created and saved in Policy Author, but are not yet deployed or scheduled for deployment. This tab is most useful for policy devel­opers as they review and refine the definitions of policies, in preparing them for deployment. This is most useful in organizations where the responsibilities for initially constructing policies, and for reviewing and finally deploying them, have been separated among different administrators. In such cases, this tab will be useful for limiting the display to only objects that have yet to be approved and deployed.

When evaluating draft components, remember to make use of the Preview tab, in the Details pane. This tab lists the actual entities that would be covered by the component as defined; this list is useful for catching errors in component definitions, which might include entities you do not intend, or exclude ones you need to cover with the component. The Preview feature is especially useful when validating User components. Preview is available only for components, not policies.

Since either the deployment itself or any adjustment of an object’s definition is done in Policy Author rather than here in Policy Manager, the Open command is useful when reviewing Draft objects. For any object in the grid, the user can right-click and select Open, and Policy Studio toggles to the Policy Author view, with that object open and ready to edit or deploy.

Monitoring Pending Objects

Once an object has been submitted for deployment, it appears in the Pending tab. More precisely, this tab shows policies and their child components in the selected folder that are in either of the following pending states:

·        Objects that have been created and saved and scheduled for deployment, but are not yet actively deployed because the scheduled deployment time has not yet arrived

·        Objects that have been marked for deactivation, but are still in Pending Deactivation state.

This tab is most useful for policy administrators as they conduct a final review of all policies that have been submitted for deployment. It is also useful for trou­bleshooting, as for example in discovering policies that administrators expected to be active but are not, since they are still pending deployment.

To perform this kind of troubleshooting:

  1. Select the Pending tab and look in the Object grid for the policy that you expect to be deployed.

  2. If you find it here, check the time displayed in the Submitted column; if it is within an acceptable time frame, you might just leave it as is, to deploy automatically.

  3. If the displayed time is too far in the past, you can select the object, right-click, and select the Open command to open the object in Policy Author.

  4. From here, you can deploy the policy by clicking the Deploy icon or select­ing Actions > Deploy.

If you expected an object to be deployed but it is not, and it does not display on this tab, first check the Draft tab—maybe it was never deployed at all. If you do not see it there, go back to the object in Policy Author and check the life cycle bar to confirm its current status.

Monitoring Deployed Objects

The Deployed state tab shows objects in the selected folder that are currently deployed and active on one or more enforcers in the system. This tab is most useful for finding out where a given object is deployed, and for viewing the def­initions of currently deployed objects. If you have deployed an object for the current time, this is where you look to confirm that it has been deployed.

As on any other tab, the Details pane in the lower right provides details of the definition of whatever object is selected in the Object Grid, above it. On this tab, however, the Details pane provides a special tab, Hosts, which displays information on all the enforcer hosts where the selected object is currently deployed. This information includes the name of the host, the user who deployed the object to that host, and the type of enforcer running there. This tab displays in the Details pane only when the Deployed state tab is active.

Monitoring Deactivated Objects

The Deactivated state tab shows policies and their child components in the selected folder that have been actively deployed at some point, but have since been deactivated. This tab is most useful for policy administrators or designers who are authorized to redeploy earlier versions of policies, or who are trouble­shooting policy conflicts or other problems that may have been caused by improperly defined policies.

Another common use for this tab is to confirm that a policy or component you expect to be deactivated, really is in that state. (This doesn’t constitute moni­toring, exactly; it’s more like confirmation of a previous action.)

Monitoring All Objects

The All tab lets you view all policies and components in the selected folder (and sub-folders), regardless of what state they are in. In this sense it resembles the whole object tree as displayed in Policy Author.

This tab provides the broadest view of all policies and components, and is most useful for any administrator who is responsible for the larger picture—for exam­ple, who need to view many policies at once, regardless of deployment status, to check for conflicts between those yet to be deployed, and those pending deployment. Compared to viewing objects in Policy Author, it offers the benefit of viewing definition details more quickly, since you can do this by simply select­ing objects in the Object grid, which displays the details in the Details pane. This is faster and more convenient that viewing full definitions in Policy Author, which requires that you open each object before you can view its definition.

In a well-developed implementation, the All tab may display such a large num­ber of objects that it becomes unwieldy, and it may be preferable to filter by one of the other state tabs. Because the All tab displays an unfiltered view of all objects, the Search tool becomes especially handy when using it.

This tab is useful when you are interested in some component or policy, but do not know what state it is in. In such cases you can start at the All tab, find the object (using the Search tool, if necessary), and then move to the appropriate state tab once you know what state the object is in.

Using Versions

The version mechanism is useful in keeping track of changing objects whose def­initions evolve, so you can see the definitions of previously deployed policies. This is most useful for auditing purposes, in cases when you need to demon­strate that a certain set of policy controls were in effect at a certain time.

Each newly created object, in Draft state, starts as Version 0. Once it is deployed, it becomes Version 1. Thereafter, every time the object moves out of Deployed state and is then redeployed, its version increments by one. Note that the version number increments simply by virtue of changing back into Deployed state, regardless of whether any changes have been made to the definition of the object. A date stamp is permanently assigned to each incremental revision, which provides a useful tool for managing and keeping track of versions.

When working with versions, it is important to understand the distinction between an object’s state and its status, as discussed below. Different versions of the same object can be in different statuses, and in such cases the same object will be listed in more than one row in the Object Grid. In addition, if an object has undergone several changes, the different versions of it will be listed in the grid, with the same status—for example, all obsolete versions would be listed on the Deactivated tab, showing Inactive status.

About Object Status

As mentioned previously, it is helpful to think of the state tabs as filters, which let you view all objects in one of the possible states at a time. In addition, though, the Object grid on each tab has a column labeled Status, which reflects something a bit different. For this reason, a given state tab may list objects in more than one status. For example, objects listed on the Deployed tab may have a Deployed status, or they might have a Pending Deactivation status—they are currently deployed, but can be pending deactivation because the actual deactivation has not occurred yet. Simply put, an object’s status may be thought of as a sub-category of its state at a given time.

The following table lists and defines all possible statuses for the four possible states. Note that these statuses are the same ones displayed in the lifecycle control bar, at the bottom of the Policy Author window.
