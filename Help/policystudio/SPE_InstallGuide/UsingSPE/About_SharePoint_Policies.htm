About SharePoint Policies

As is discussed in the introduction, the Entitlement Manager for SharePoint supports four kinds of policies. Each of these is discussed in more detail below, along with example use cases for each kind of policy:

1       Classification Policies

2       Access Control Policies

Classification Policies

Policies can apply labels to documents as they are uploaded into SharePoint. After documents are classified, Access Control and other policies can reference these tags. Classification can occur automatically (through content analysis for example), or can be user-specified (where a user selects the appropriate tag). Classifications can also be applied to existing files already uploaded to SharePoint, through batch processes that apply classifications to all documents in a designated location at once (based on content analysis, the location where the file resides, document properties, and more).

Example Use Cases

Set up content aware tagging so that whenever a document is uploaded to a SharePoint site that contains the word "ITAR," an "ITAR" label will be assigned in the "Export Security" column.

For a target site collection, run a nightly batch process that performs content analysis of all new or updated documents. Documents that include the word "restricted" are tagged "restricted."

Note: For more information on example policies, see Designing Classification Policies.

Access Control Policies

There are four kinds of Access Control policies: (1) block or allow access to content, including the creation of new webparts (2) control access to the ability to change or inherit site settings and permissions, and (3) add view filtering on top of existing access control policies.

(1) Control Access and Creation

For policies that block or allow access to documents, you can define a SharePoint portal as a file location: the various sites, libraries, document lists, etc., are analogous to directories, whose content you want to control in certain ways. Access Control policies either allow access to a portal location or deny it, for either reading or writing purposes, or both.

Policy designers can also control the creation of SharePoint webparts (lists, libraries and so on). This functionality provides a solution to one of the most difficult control problems of SharePoint: limiting how individual users can share data resources by opening new lists, to which they have control over access by others. Another common problem is the inefficient use of company resources (for example, where users create sites that no one ever actually uses or that duplicate existing sites), wasting storage space and confusing end users. Policy designers can limit or monitor (see Audit Policies below) the creation of the following kinds of SharePoint structures: Document, Libraries, Announcements, Project Tasks, Form Libraries, Contacts, Issue Tracking, Wiki Page Libraries, Discussion Boards, Surveys, Picture Libraries, Links, Custom Lists, Slide Libraries, Calendars, Custom List in Datasheet Views, Translation Management Libraries, Tasks, Languages and Translators, and Imported Spreadsheets.

(2) View Filtering (Security Trimming)

In addition to allowing or blocking access, the Entitlement Manager can allow or block the viewing of content. Security trimming comes in three forms: Webpart trimming controls which webparts display when a user views sites in SharePoint, List Trimming controls which items in a list display, and Search Trimming determines which search results display when a user runs a search within SharePoint.

(3) Control Site Settings Permissions

Another kind of Access Control deals with how SharePoint users access and modify site settings and permissions. For instance, consider the case where a SharePoint Administrator extends site access to users that should be unauthorized, according to company policies. In this case, SharePoint’s discretionary permissioning model can lead to unauthorized access. The solution to this problem lies in policies that grant users permission to access and use site content, but deny access to the administrative settings for that content. These kinds of policies are referred to as SharePoint settings policies.

Entitlement Manager for SharePoint policies can also control how permissions and other settings are inherited between sites. For example, your SharePoint portal may have a Customer Sites area, which contains one document library for each of your customers. The customer libraries may inherit their permission levels from the parent area, Customer Sites. However, one customer’s data may be much more sensitive than another’s, and you may not want every user with access to some sensitive customer site (thanks to the parent permissions settings) to be able to change that site’s settings, which might to allow inappropriate users access to it. This kinds of over-permissioning is a common problem caused by SharePoint’s discretionary model.

Example Use Cases

Allow only members of the design team working for Client A to access any documents on Client A’s SharePoint site. The individual users belonging to the authorized design team may change over the project lifecycle, but the control over client data will be maintained.

Prevent any designer from uploading documents to any client’s site that are not tagged as belonging to that client (including those not tagged at all). This prevents exposing one client’s data to another client.

Prevent any user from moving documents from any client-related SharePoint site to any publicly open site, and from downloading to any open file server location.

Prevent users who have regular access to ITAR (export controlled data) from creating new lists. This helps enforce a companion policy that ITAR data only be stored in the appropriate locations.

For classified data, prevent unauthorized users from viewing a team list (or other webpart) that contains data they are not authorized to view. When users conduct searches, search results should also include only documents users are authorized to access.

For all sites that contain ITAR (export controlled) data, block all users except a designated Administrator from modifying Administra­tive settings for sites. This prevents even site owners from extending access to unauthorized users.

Note: For more information and example policies, see Designing Access Control Policies, Adding Security Trimming to Access Control Policies, and Example Policy: Access Control for SharePoint Administrative Settings.

Data Protection Policies

Data Protection policies ensure that data will be persistently protected even after documents are downloaded from the SharePoint server. This occurs through the tagging of data with persistent metatags that can be references by policies enforced by other NextLabs products. Tag on download policies can apply different kinds of tags to documents (depending on how they’re designed), either to reference a specific column, multiple columns, or a user-defined value.

Example Use Case

Set up automatic file tagging whenever a document is downloaded from a Project Team’s SharePoint site, marking that site as its source. This allows other local policies to limit distribution of that document to other Project Team members.  

Note: For more information on example policies, see Designing Data Protection Policies.

Audit Policies

It is important that administrators have insight into what end users are doing in SharePoint, including how end users are creating new areas on sites, what resources they are posting there, and what permissions they are changing. Audit policies allow you to log these details, as well as include notification e-mails to administrators in appropriately serious cases.

Example Use Cases

Whenever anyone creates any new site, or a new site within some specified other site, log the event details

Whenever anyone deletes any site, or a new site within some speci­fied other site, log the event details and send a notification e-mail to an administrator

Whenever any user performs any action on the site, log the details (would use ANY action)

Whenever anyone accesses the portal from a host outside the com­pany, log the event details (would use ANY action, and ANY user on a Computer NOT IN the Company location site)

Note: With these policies deployed, Administrators could run reports that show all new sites created, all sites deleted, or all site access, each of which can be filtered by department, user, and activity. You can also run lifecycle activity reports, to identify unused or barely used sites.

Note: For more information on example policies, see Designing Audit Policies.
