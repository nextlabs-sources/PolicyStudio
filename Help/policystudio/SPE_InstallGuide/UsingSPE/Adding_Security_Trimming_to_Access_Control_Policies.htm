Adding Security Trimming to Access Control Policies

Security Trimming refers to the Entitlement Manager for SharePoint’s ability to block the display of any content the current user does not have access to, according to some currently deployed policy. When this feature is enabled, any user who is blocked from using certain documents posted on SharePoint not only cannot download or open them, but also cannot even see they are posted.

Depending on how the system is configured, this can be applied at various levels: for whole webparts (sites, lists, libraries, calendars, wikis, tabs, etc.), individual items in lists, and even individual items returned by the site search tool. The required software components are automatically installed on each SharePoint server where an enforcer is installed. However, the feature is disabled until an administrator manually turns it on. This section describes how to enable the various kinds of Security Trimming. Each kind of trimming is enabled separately, and is discussed as separate procedures in this section:

1       Webpart Trimming

2       List Trimming

3       Search Result Trimming

You do not create Security Trimming policies the same way you do for other kinds of Access Control. Instead, you create Portal components that represent where the trimming will be applied, and then enable the kind of Security Trimming you want to apply. Keep in mind that each kind of trimming is enabled individually, so you can enable all of them, or just one of them, depending on your needs. In other words, you can add List Trimming to restrict viewing of content for one list that is targeted in policy, you can add Webpart Trimming to restrict a different SharePoint resource targeted in policy, and enable Search Result Trimming for all resources targeted in policy, if desired.

Webpart Trimming

Webpart Trimming allows policy designers to expose individual webparts to specified users as appropriate, while hiding webparts from users who are not authorized to use them. In SharePoint terminology, webparts include tables, lists, document libraries, directories, calendars, filters, and many other tools such as dashboards and search fields. This feature is supported by Entitlement Manager for SharePoint, release 5.5 and higher.

Webpart Trimming is either enabled or disabled for the whole site collection, and when enabled will work the same way for all component structures (sites, pages, lists, etc.) of that site collection. It can be turned on or off at any time. When it is disabled, any policies that rely on webpart-level granularity will not work properly.

The reason Webpart Trimming is designed to be explicitly enabled by an administrator is that it works by adding a special control to every master page in the site collection. Some SharePoint administrators may consider this an unacceptably intrusive option, or may want to spend some time considering whether it is or not; in such cases, they have the option of leaving it turned off.

Strategies for Designing Security Trimming Components

As with any other kind of component, you can group various kinds of webparts together in one component based on whatever is useful. In many cases, this will mean defining components based on organizational ownership, which map quite closely between site structure and users—for example, a component called All Marketing Lists, which might be exposed only to users in the Marketing group.

However, Webpart Trimming also frees you to define components by any other criterion you like, whether or not it reflects the structure of the SharePoint sites. For example, you could define a component based on the sensitivity of data—Highly Confidential Projects—which could only be exposed only to authorized users, without regard to where the projects are located across various sites in the site collection.

This ability to define components that cut across the structure of the site provides far more granular and flexible control possibilities than native SharePoint access controls by themselves.

List Trimming

List Trimming lets you design policies that hide individual items in lists from users who are not authorized to use the resources they are linked to. As with Webpart Trimming, the links enforced by List Trimming policies must be defined as portal resource components based on their URLs.

Note: For more information on creating Portal resource components for trim­ming, see Creating Portal Components for Security Trimming.

For example, your site might have a document library containing project design documents, some of which are highly sensitive and others less so. Because you want all users to have read access to the less sensitive documents, you cannot use policies based on Webpart Trimming here. However, you can use List Trimming to expose the individual links to more sensitive documents only for authorized users—the Advanced Projects team—while hiding them from users not on that team.

Also like Webpart Trimming, List Trimming can be enabled or disabled for the whole site collection, and when enabled will work the same way for all component structures (sites, pages, lists, etc.) of that site collection. The procedures for activating List Trimming across all the lists in a site collection are described here: Enabling Webpart and (Global) List Trimming.

It can be turned on or off at any time. When it is disabled, any policies that rely on link-level granularity will not work properly. List Trimming can also be enabled for individual lists, as is discussed below.

Search Result Trimming

The third type of trimming applies to the results of searches conducted within SharePoint portals. Search Result Trimming allows you to limit displayed search results to only those web parts or documents (i.e., list items) which the searcher is authorized to view. For example, suppose a user searches a site collection for the string “Cyberdyne Corp.” If some policy blocks that user from viewing, opening, or using documents belonging to or associated with Cyberdyne Corp, then any result involving Cyberdyne documents that appears in the search results will be trimmed—that is, will not be displayed to that specific user.

Search Result Trimming is closely integrated with SharePoint’s crawl rule mechanism. The procedures are slightly different in SharePoint Server 2007 and 2010, so they are discussed in separate sections below.

About SharePoint Crawl Rules

The internal administrative tools in SharePoint allow you to define crawl rules, which are essentially filters that control what paths will be included or excluded from content searches initiated by SharePoint end users. It is helpful to think of these rules as a primary filter for search results, while the search result-related policies you define in Compliant Enterprise provide a secondary, more granular filter.

In practice, this means that any existing crawl rules will block users from searching on entire sites or some branches of sites, based on designated starting point URLs. Content on site locations already blocked by crawl rules will not be exposed to users regardless of whether they are allowed or blocked by Compliant Enterprise policies, since they are already blocked by crawl rules, and these have a higher priority. For site locations not blocked by crawl rules, Compliant Enterprise policies will then be applied to users (in a much finer and more context-aware way), to display or hide specific search results pointing to those locations. In short, Compliant Enterprise policies are only enforced on site locations that are not already restricted by crawl rules.

Note that you can define crawl rules to cover both SharePoint server and file server shares, which means that you can set up search result trimming to cover searches of both SharePoint sites and file shares.

Uninstalling Search Result Trimming (SharePoint 2007 and 2010)

If you are using search result trimming and need to uninstall your Entitlement Manager for SharePoints for an upgrade, you must manually disable search result trimming before you perform the uninstall. To do this, use the Entitlement Manager for Administrator utility and run the following command:

ce_spadmin -o searchresulttrimming -uninstall

Using the Entitlement Manager for SharePoint Administrator

As an alternative to controlling trimming features through SharePoint’s own administrative pages, you can use the special utility called Entitlement Manager for SharePoint Administrator. This utility is automatically installed on each Entitlement Manager for SharePoint host, in the C:\Program Files\NextLabs\SharePoint Enforcer\bin directory. It is available only to users with administrative permissions for the site collection, based on their Windows logon. The procedures for running the utility are the same for SharePoint 2007 and 2010.

Controlling Security Trimming

You can use the Entitlement Manager for SharePoint Administrator to enable security trimming. This utility, which is automatically installed on each Entitlement Manager for SharePoint host, can be used only by users with administrative permissions for the site collection, based on their Windows logon.

Follow these steps to use the Entitlement Manager for SharePoint Administrator utility to enable security trimming:

  1. Open a Windows command prompt window on the SharePoint Server host.

  2. Set the environmental variable to give the path for the administrator util­ity:

    3. C:\Program Files\NextLabs\SharePoint Enforcer\bin

  3. Type the Entitlement Manager for SharePoint Administrator command:

ce_spadmin -o securitytrimming -enable -url <URL>

where <URL> is the fully qualified URL for the site collection, e.g. http:\\sharepoint.cyberdyne.com. This element is optional; if it is not specified, the command applies to all SharePoint site collections defined on the server or server farm.

Disabling Security Trimming

Once this feature has been enabled, you can use the same utility to disable it. To do this, set the same environmental variable and use the command:

ce_spadmin -o securitytrimming -disable -url <URL>
