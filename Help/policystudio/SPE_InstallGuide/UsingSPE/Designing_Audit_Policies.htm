Designing Audit Policies

The last family of policies that you can design for the Entitlement Manager for SharePoint includes audit policies. This section includes the following information:

1       Example Policy:  Event Auditing

2       Example Policy:   Permission Change Auditing

Example Policy:  Event Auditing

Anytime a policy decision is enforced, the event is automatically "logged" and available for reporting. This includes Classification, Access Control, and Data Protection policies. However, there may be times when you want to log events without intervening in user behavior or applying classifications to data. By simply creating "Monitor" policies, you can collect information about any user event for which there is a SharePoint action (for a list of available actions, see What Actions Can Be Enforced?).

A common use case would be where Administrators need to monitor the creation of new webparts within SharePoint. SharePoint’s benefit in allowing users to share information by easily creating new sites, lists, and libraries is one of the main reasons SharePoint is so popular. However, many organizations seek to monitor (not block or necessarily prevent) the creation of new webparts to ensure that data is not being duplicated unnecessarily (misallocating company resources). Note that this is a different use case than an Access Control policy, which might prevent certain SharePoint users from creating new lists containing restricted data. (For more example use cases, see the overview of Auditing Policies.)

Example Policy:   Permission Change Auditing

One source of the difficulty of managing SharePoint entitlements is the fact that owners of individual sites or web parts can grant new access permissions to other users, beyond the knowledge or control of the central SharePoint administrator. As a useful tool in monitoring and controlling this kind of activity, the Entitlement Manager for SharePoint lets you deploy policies that log all detailed information about all instances where end users change permission levels for sites they control. You can then use this information to run audits of who is changing permission levels, in order to track what changes are being made

Permission Change Policies

The permission change policies you deploy should have these characteristics (each shown in the figure below):

A.They use the Monitor enforcement effect.

B.They include All Users as subject.

C.They use an Edit action component, which need include only the Cre­ate/Edit basic action.

D.As a resource, they use a SharePoint settings component, as described in the previous section. These are Portal-type resource components that represent a specific group of administrative set­tings for a specific site (see Defining SharePoint Settings Components).

E.They must include the Log Permission Change custom obligation.

 

Sample Permission Auditing Policy

Once you construct and deploy such policies, the Entitlement Manager for SharePoint begins logging detailed information on all instances of users editing the administrative settings specified in the policy.
