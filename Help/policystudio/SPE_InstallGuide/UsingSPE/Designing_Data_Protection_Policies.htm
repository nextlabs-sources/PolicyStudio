Designing Data Protection Policies

Data Protection policies refer to the capability to extend protections to data even when data is download or exported outside of SharePoint. This section provides the following information about these policies:

About Download Tagging

You can do this by designing policies that automatically assign metadata tags to documents when they are downloaded from SharePoint, so that other NextLabs enforcer policies can reference these tags wherever they move throughout your enterprise. These tags can be applied to documents listed in document libraries and generic document lists. Tags may be based on the name of the column in which they are listed, or defined however the policy designer wishes.

For example, a document library on some site may have a column for "Status" to let users track whether a document is in Draft, Pending, or Approved state. You might design a policy that lets authorized users download Draft documents, but blocks them from distributing them as e-mail or IM attachments. To make this policy work, you would set up automatic tagging so that all documents on this site are tagged with a “State= [  ]” property when they are downloaded. Because the tag would persist wherever the document is copied, your desktop-level distribution policy will prevent it from being e-mailed even though it has left the protection of the Entitlement Manager for SharePoint. Extending our engineering site collection use case, another scenario may be that you want the "IP" tag on upload value to be persistent so that even engineers who are authorized to download these documents may not email them to unauthorized users.

When a user downloads a document from SharePoint Server, uploads it to a different location on SharePoint Server, then downloads it again, the values of the tags will be based on its last location on a SharePoint server. Any tags that are based on any prior locations on any SharePoint server will not be removed. Any tags that are based on any prior locations on a non-SharePoint server (i.e. tags that were created by a Tagging Policy Assistant), will not be removed.

Supported Download Tagging Methods

This feature lets you automate four kinds of SharePoint download tagging, each of which corresponds with a custom obligation that must be configured:

4       System-Column Tagging: Applies tags that reflect all the standard sys­tem columns present by default in SharePoint lists and libraries. These include Resource, Signature, Modified By, File Size, Type, Sub-type, Cre­ated, Modified, Name, and Created By.

5       All-Column Tagging: Applies tags that reflect all system columns plus all columns marked as being Displayed in the specified document library or list, as long as their data type is Text, Number, Currency, or Choice. (You mark columns as Displayed by checking the Display checkbox in the Edit View screen of the library or list.) That is, columns with other data types, and those not marked for display, will not be covered by All-Col­umn tagging policies.

6       Specific-Column Tagging: Applies tags that reflect only a specific subset of columns. These are defined within individual policies. These can be different for different policies that may be applied to the same set of SharePoint resources. They may be system or custom columns, or a com­bination of both.

7       User-Defined Tagging: Allows the policy designer to define the tags that will be applied. These can be any alphanumeric text string you like, and may be completely unrelated to the column names in the SharePoint list or library.

Required Configuration

Example Policy: Persistent Tag on Download  

You determine what kind of download tagging should occur based on the custom obligations appropriate to your needs. When you design policies of this kind, you need to focus on the following questions:

  1. Which SharePoint document libraries or lists contain documents that I am concerned about tagging?

  2. What columns (that is, properties) in these libraries or lists do I want to use as tags when documents are downloaded? Do I need to add any new columns—Security Level, for example—just for the purpose of tagging?

  3. Given the properties I want to use as tags, which of the four types of tag­ging do I want to use for this policy?

Once you have done this planning, you can construct policies to implement automatic tagging. Bear in mind the following guidelines:

10   The policy must be Document type.

11   The resource involved must be a Portal Content component, typically based on the URL of the SharePoint page containing the library or list you are targetting. This could be more than one URL, as long as you are tagging based on columns present in all lists or libraries. You can even use wildcards to represent “any library or list." For example, the policy might tag any document from any SharePoint page that has an Internal Only column. Such a policy would tag documents downloaded from lists that have this column defined, but would not effect files downloaded from lists that do not have this column.

12   The action must be Open.

13   The policy must include one of the SharePoint tagging obligations. If you use any of the three column-based obligations (All-Column Tagging, User-Defined Tagging, or Specific-Column Tagging, as shown below), you may also need to specify which column you want to use.
