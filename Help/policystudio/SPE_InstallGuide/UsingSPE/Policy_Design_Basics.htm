What Actions Can Be Enforced?

Like all Entitlement Managers, the Entitlement Manager for SharePoint can enforce against a finite set of user actions. Certain actions can only be used in policies of a specific type. This is described in the following table.

Basic Action

Description

For Use in Policy Types

Attach to Item

Attach a document to an item in a portal list.

Access Control, Audit

Create/Edit

Create a new file or item, rename, or change the contents of an existing one. Specific actions include:

  • Create a new document

  • Edit the content of an existing document

  • Overwrite an existing document by any means: copying a file with the same name from another location, renaming another file, using the Save As or Export com­mand in another file, etc.

  • Create a portal site, page, list, library, library item, or column

  • Edit a portal site, page, list, library, library item, col­umn, or site property by any means (in datasheet, in spreadsheet, etc.)

Access Control, Audit

Delete

Permanently remove a file or portal item from storage. Specific actions include:

  • Delete a file

  • Delete a site, workspace, structure, list, or library item

Access Control, Audit

Download

Download a file from a SharePoint site. This is a special action, meaning it must be configured before it is available for use (see Configuring Special Actions).

Access Control, Audit

Export

Export a portal item. (This is a portal-specific basic action.) This action covers two particular cases:

  • Export list to datasheet

  • Export library to spreadsheet

Access Control, Audit

Move

Delete a file from its current storage location and place it in a different location. Specific actions include:

  • Move a file, within the file system

  • Move a site, workspace, structure, list, or library item, within the portal server

Access Control, Audit

Open

Open a file or portal item for viewing. Specific actions include:

  • Open a file

  • Access a site, workspace, structure, list, or library item on a portal

  • Download an item from a portal

  • Upload an item to a portal

Access Control, Data Protection, Audit

Upload

Upload a file to a SharePoint site. This is a special action, meaning it must be configured before it is available for use (see Configuring Special Actions).

Access Control, Classification, Audit

How Can Resources Be Targeted?

The Entitlement Manager for SharePoint allows designers to define resources in highly flexible ways. This section introduces you to portal components and describes different ways you can target resources when you design portal components.

About Portal Components

In all SharePoint-related policies, the resource component will be a Portal Content type, rather than Document type. When you create a new Portal Content component, you have two areas where you can define it: the members field and the Properties field. Both start out empty in a newly created component, as shown in the following figure.

 

Newly Created SharePoint Component

In defining a portal component, there are the following basic components. These can be used independently, or in combination, as described below.

Defined by Portal Location

If the component refers simply to some to location on the SharePoint Server—for example in the Design Documents Library on your company’s Engineering site—then the portal resource should be defined as the URL to the SharePoint location, in the Members, Portals In (or Not In) field, as shown in the figure below.

 

SharePoint Component Defined by Location Path Only

You can type the URL directly in the Members field, or you can open your browser, navigate to the SharePoint resource you want to control, copy the URL from your browser’s address field and paste it into the Members field of the policy.

Defined by Properties

A component can also be designed to represent some specific class of documents regardless of where they are stored on the SharePoint server. In this case, do not supply a location in the Members field, but select one or more specified properties under the With Properties field, as shown in the figure below.

 

Component Defined by Document Property Only

Note there are limitations on what kinds of properties are available, depending on what kind of SharePoint content or structures your component represents. Existing document resources stored on SharePoint can make full use of the properties those resources possess. Components representing site structures
(i.e., web parts) also can be defined by a wide range of properties, so long as they are to be used in policies that control access to already defined web parts. However, in defining components representing lists or libraries that will be used in policies that block their creation, only the properties Type and Sub-type may be used.

Defined by Content

You can create content aware policies using the With Content drop down menus. Content analysis can either scan content itself (as in the example below) or look for content types (credit card or social security numbers, etc.)

Combining Locations, Properties, and/or Content

Keep in mind that you can design portal components by combining locations, property, and content, to target resources in highly granular ways. Using SharePoint User Groups

For the user component in SharePoint policies, you have the option of using SharePoint groups, as long as you have enrolled them into your Policy Server dictionary. (For more on this, see the Administrator’s Guide.) SharePoint groups are those defined in SharePoint to govern access to various areas of the portal, as opposed to the groups defined in Active Directory for the Windows network in general. As a rule, it is more reliable to use SharePoint groups rather than AD groups in policies that are aimed at SharePoint resources.
